## Итерация 3 — страница новеллы + читалка + 3+3 киллер-фичи

### Страница новеллы `/novel/[id]` — новое
- `FirstChapterPreview` — карточка с первым абзацем первой главы и кнопкой «Дочитать» *(киллер #1)*
- `SimilarByReaders` — рекомендации через RPC `get_similar_novels_by_readers` *(киллер #2)*
- `ReleasePace` — столбчатая диаграмма выхода глав за 90 дней + прогноз темпа/дня недели *(киллер #3)*
- Бейдж «~8 ч» рядом с жанровыми note
- Fallback «Похожее от автора» если коллаборативка пустая

### Читалка `/novel/[id]/[chapterNum]`
Перенесли из tene `ChapterReader.jsx`:
- Шрифты: Inter, Lora, Merriweather, Roboto, OpenDyslexic
- Размер шрифта, высота строки, выравнивание, красная строка, отступ абзацев
- Настройки сохраняются в `localStorage` (`chaptify-reader-settings`)
- Прогресс чтения: `localStorage` + `profiles.last_read` (paragraphIndex + chapterId)
- Автовосстановление позиции при повторном заходе
- Горячие клавиши: +/−, F (фокус)
- Буквица на первом абзаце главы

### Киллер-фичи читалки

1. **Фокус-режим** (`src/components/ReaderSettings.tsx` + CSS в `ReaderContent.tsx`)
   Затемняет все абзацы кроме текущего (тот, что ближе к середине вьюпорта). Scroll → смещается активный абзац. Переключается горячей клавишей F.

2. **Умные закладки с цитатами** (`src/components/QuoteBubble.tsx`)
   Выделил текст мышью/тапом → появляется всплывающая кнопка «⊹ Сохранить цитату». Сохраняет в `user_quotes` (таблица с RLS). Коллекция цитат лежит в профиле — можно перечитывать любимое без перечитывания главы.

3. **Таймер сна** (`src/components/SleepTimerOverlay.tsx`)
   Пресеты 15/30/45/60 мин в настройках. По истечении — мягкий overlay «Пора сделать паузу» с кнопками «Ещё 15 минут» или «Спокойной ночи».

### SQL миграция 003 — `migrations/003_quotes_recommendations.sql`
- Таблица `user_quotes` (RLS включён, select/insert/delete только своё)
- RPC `get_similar_novels_by_readers(p_novel_id, p_limit)` — коллаборативная фильтрация
- RPC `get_release_pace(p_novel_id, p_days)` — темп выхода по дням

## Итерация 4 — админка: добавление новелл и глав + 3 киллер-фичи для переводчика

### Сделано

**Шапка (SiteHeader)** — кнопки по ролям:
- Гость: `Войти`
- `user`: `Стать переводчиком` + `Профиль`
- `translator`/`admin`: `+ Новелла`, `Админка`, `Профиль`

**Middleware** — упрощена логика беты:
- `/translator/apply` доступна любому залогиненному
- Остальное — только translator/admin; не-переводчики → `/translator/apply` (раньше был `/beta`, теперь отдельная страница не нужна)

### Новые страницы

- `/translator/apply` — форма заявки. Поля: мотивация (20–2000 симв), портфолио, языки (KR/CN/JP/EN), желаемый slug. Показывает статус существующей заявки («pending», «rejected» с комментарием модератора).
- `/admin` — дашборд:
  - Список «Мои новеллы» (для translator) или «Все новеллы» (для admin). Каждая строка: обложка, название, счётчики, статус модерации, кнопки `+ Глава` и `Редактировать`.
  - Секция «Заявки в переводчики» (видна админу): мотивация, slug, языки, кнопки «Одобрить»/«Отклонить» → вызывают RPC `approve_translator_application` / `reject_translator_application`.
- `/admin/novels/new` — создание новеллы.
- `/admin/novels/[id]/edit` — редактирование новеллы + глоссарий на той же странице.
- `/admin/novels/[id]/chapters/new` — создание главы.
- `/admin/novels/[id]/chapters/[chapterNum]/edit` — редактирование главы.

### Страница новеллы — владельцу показываются кнопки
- `+ Добавить главу`, `Редактировать` рядом с «Читать»
- `Править` на каждой строке главы в списке

### Киллер-фичи для переводчика

1. **Глоссарий проекта** — `src/components/admin/GlossaryPanel.tsx` + таблица `novel_glossaries`
   Словарь `термин оригинала → перевод` с категорией (персонаж/место/термин/техника) и примечанием. В форме главы:
   - Сайдбар со списком первых 10 терминов
   - В live-preview pane каждое совпадение с глоссарием подсвечено `<mark class="glossary-match">` (жёлтая подложка), на hover — перевод в title
   - Счётчик «Совпадений с глоссарием: N» в заголовке preview
   
   В будущем: читатель в главе сможет нажать на подсвеченный термин и увидеть объяснение.

2. **Автосохранение черновика** — `src/components/admin/ChapterForm.tsx` + `src/components/admin/DraftBanner.tsx` + таблица `chapter_drafts`
   При изменении content или chapter_number → debounced 2 сек → upsert в `chapter_drafts` (UNIQUE user_id+novel_id+chapter_number). Индикатор «✓ Черновик сохранён». При открытии формы, если для этой новеллы есть черновик — показывается `DraftBanner` с кнопками «Восстановить» / «Начать заново».

3. **Статистика главы в реальном времени** — `src/components/admin/ChapterStats.tsx` + `computeChapterStats()` в `src/lib/admin.ts`
   Сайдбар, обновляется при каждом изменении текста:
   - Слова, знаки, абзацы, расчётное время чтения (180 wpm)
   - Топ-6 повторов (исключая стоп-слова из ~100 русских/английских)
   - Длинные предложения (30+ слов) — с алёртом «Возможно стоит разбить»

## Итерация 7 — «Моя библиотека» + рабочая кнопка закладок

### Новая страница `/bookmarks`
Все закладки пользователя в одном месте, без новых таблиц в БД — всё считается из `profiles.bookmarks` + `profiles.last_read`.

Киллер-фичи:
1. **Умные категории** — 5 вкладок, автораспределение по данным:
   - **Читаю** — last_read не старше 14 дней, прогресс < 100%
   - **На паузе** — last_read 14–90 дней назад, прогресс < 90%
   - **В планах** — в закладках, но не открывал
   - **Прочитано** — прогресс ≥ 90% или новелла завершена + прогресс ≥ 80%
   - **Брошено** — last_read > 90 дней, прогресс < 50%
2. **Темп чтения** — для «На паузе»/«Брошено» карточка считает оставшиеся главы и даёт прогноз: «По 2 гл/день — 27 дней. По 5 гл/день — 11 дней».

### Контекст
Поскольку chaptify и tene **живут на одной БД Supabase**, отдельного переноса аккаунтов не требуется: один раз войдя на chaptify под тем же Telegram/email, пользователь автоматически попадает в свой tene-аккаунт со всеми закладками, монетами, подписками, прогрессом чтения.

### Методы входа

1. **Telegram Login Widget** — `src/components/auth/TelegramLoginWidget.tsx`. ## Итерация 13 — центр уведомлений

### Новая страница `/notifications`
Таблица `public.notifications` уже существовала в tene — расширил её и добавил автоматическое наполнение через триггеры. Старые уведомления из tene продолжают работать как были; новые события накидываются триггерами сами.

### Автоматические события (триггеры БД)
Миграция 007 вешает триггеры, которые создают уведомления без вызова из кода:

- **Новое сообщение** → триггер `trg_notify_new_message` на `direct_messages INSERT` → уведомление получателю
- **Запрос в друзья** → триггер `trg_notify_friend_request` на `friendships INSERT` (status=pending) → addressee видит запрос
- **Запрос принят** → триггер `trg_notify_friend_accepted` на `friendships UPDATE` (pending → accepted) → уведомление отправителю
- **Ответ на комментарий** → триггер `trg_notify_comment_reply` на `comments INSERT` (reply_to NOT NULL) → уведомление автору родительского
- **Новая подписка** → триггер `trg_notify_new_subscription` на `subscriptions INSERT` (status=active) → уведомление переводчику

Каждый триггер тянет красивое имя актора (`translator_display_name ?? user_name`), формирует target_url и group_key. Никаких изменений в существующем коде tene не нужно — триггеры работают прозрачно.

- `actor_id uuid` — кто вызвал событие (для аватара)
- `group_key text` — для автогруппировки похожих событий
- `ref_novel_id bigint` — опциональная связь с новеллой

### Три киллер-фичи

1. **Автогруппировка** — RPC `list_notifications` через `DISTINCT ON (group_key)` возвращает один элемент на группу + `group_count`. В UI: «Иван написал сообщение · и ещё 4». Не засоряет ленту одинаковыми.
2. **Фильтры по типу** — табы «Все / Непрочитанные / Друзья / Сообщения / Комментарии / Подписки». Через `?filter=` в URL.
3. **Inline-кнопки** — для уведомления о запросе в друзья прямо в карточке есть «Принять» / «Отклонить» — не нужно переходить в `/friends`. Один клик — и готово.

## Итерация 16 — per-translator выплаты + FAQ

### Архитектура выплат

**Tribute — автоматически.** У Tribute есть честные вебхуки. Каждый переводчик получает свой уникальный URL `https://tene.fun/webhook/tribute/<token>`, который копирует в настройки Tribute. ### Почему был важен вопрос «кому идут монетки»

До этой итерации система жила по модели «у юзера один кошелёк, у переводчика тоже» — это действительно странно, потому что непонятно, как разделять деньги между несколькими переводчиками.

**Правильная модель (как у Patreon/Steam/Yandex Music):** деньги физически лежат у платформы, переводчикам **причитаются** виртуальные суммы, которые платформа раз в месяц закрывает реальными переводами.

Это лучше для всех:
- Читатель один раз пополняет — тратит где хочет, без повторного ввода карты
- Переводчик не регистрирует платёжки на себя лично
- Налоги централизованные
- Платформа гибко настраивает комиссии/курсы

1. **✓ Опубликовать сейчас** (по умолчанию) — `published_at = now()`
2. **⏰ Запланировать** — datetime-local, `min = now + 5 мин`; если ввели прошлое/ближайшую минуту, трактуем как «сейчас»

## Платёжная система end-to-end (актуально на 23.04.2026)

### Базовая модель: chaptify не проводит деньги через себя
Деньги читателя → переводчику напрямую через **Boosty / Tribute / VK Donut / Patreon**. Chaptify только подтверждает факт платежа, чтобы открыть платные главы.

Что у нас «своё» — внутренние **монеты**: цифровая услуга от платформы (продаются за рубли), тратятся на покупку отдельных глав или чаевые. Это не выплаты другим людям — налоговая упрощается.

### Со стороны переводчика

#### 1. Подключить способ оплаты — `/profile/settings` → блок «Способы оплаты»
Таблица `translator_payment_methods` (мигр. 037). Можно подключить несколько провайдеров. Каждый ряд:
- provider: `boosty | tribute | vk_donut | patreon | other`
- url: ссылка на свою страницу оплаты
- instructions: для читателей («оплати тир Фанат и напиши мне в ЛС»)
- enabled: вкл/выкл
- tg_chat_id: ТОЛЬКО для Boosty — id закрытого TG-чата подписчиков для авто-сверки (мигр. 038)

Старое поле `profiles.payout_boosty_url` (settings) ещё работает (бэкфилл из него попадает в `translator_payment_methods` при первой миграции).

#### 2. Tribute — webhook-автомат
Есть честный webhook от Tribute. На `/admin/payouts` копируется уникальный URL `https://tene.fun/webhook/tribute/<token>`, вставляется в Tribute. Каждый платёж → бот находит читателя по `payment_code` или `telegram_id` → начисляет монеты или активирует подписку (`provider=tribute`).

Health-индикатор «активен N мин назад» по `payout_last_tribute_event_at`.

#### 3. Boosty — три механики (по убыванию автоматизации)

**A. Полный авто-синк через TG-чат (мигр. 038)**
1. Переводчик настраивает Boosty так, что подписчики автоматически попадают в его закрытый TG-чат
2. В этот же чат приглашает `@chaptifybot` участником
3. На странице оплаты у читателя кнопка **«Я уже подписан(а) — открыть автоматически»**
4. Бот через `getChatMember(chat_id, user_tg_id)` проверяет членство → RPC `grant_subscription_from_boosty_chat` → подписка active на 1 месяц

Если читатель отписывается на Boosty → автоматически вылетает из чата → следующая попытка авто-синка не сработает. Существующая подписка живёт до `expires_at`, потом сама expired (см. ниже про отмены).

**B. Полу-авто claim через бота (PR #5 в server-репо)**
Читатель платит на Boosty. На странице оплаты получает короткий код `C-XXXXXXXX`. Идёт в `@chaptifybot`, шлёт `/claim <код>` → переводчик в личку видит inline-кнопку «Одобрить / Отклонить».

**C. Ручной claim через UI (мигр. 035)**
Если бот недоступен / читатель не в TG: на `/t/[slug]` оформляет заявку через форму. Заявка попадает в `subscription_claims` → переводчик видит на `/admin/subscribers` → одобряет / отклоняет → создаётся `subscriptions` с `plan='external_claim'`, `provider='boosty'`.

#### 4. Управление подпиской переводчиком — `/admin/subscribers`
- **Заявки** — pending claims с кнопками Одобрить / Отклонить (с причиной) / Написать в ЛС
- **Активные подписчики** — все active subscriptions с expires_at; кнопка **⊗ Отозвать** (мигр. 041) для случая «оплатил, потом отписался на Boosty», даёт причину в push читателю
- **История** — последние 30 разобранных заявок

#### 5. Учёт выплат — `/admin/payouts` + `/admin/all-payouts` (только админ)
- payout_cycles (мигр. 011): монеты, заработанные переводчиком, → ₽ → выплачены такого-то числа таким-то способом
- Дашборд /admin/all-payouts: селектор месяца, настраиваемый курс и комиссия, CSV-экспорт, кнопка «Выплачено ✓» по каждой строке

### Что происходит после отмены на Boosty?

**Короткий ответ: подписка живёт до `expires_at` и потом естественно истекает.**

Подробнее:
- При одобрении claim или авто-синке `expires_at = now() + N месяцев` (по умолчанию 1)
- Chaptify **не видит** факта отмены на Boosty (нет API)
- Читатель пользуется подпиской до её истечения — это честно, он за этот период заплатил
- После `expires_at` `can_read_chapter` начнёт выдавать `false` для платных глав
- Если переводчик хочет отозвать подписку досрочно (например, узнал про cancel из личных сообщений Boosty) — кнопка **⊗ Отозвать** на `/admin/subscribers` (мигр. 041)
- Авто-синк через TG-чат (вариант A) НЕ ре-проверяет существующие подписки. Делает grant только когда читатель сам нажимает «Я уже подписан(а)». Если читатель не нажимает — мы не узнаём что он отписался. Можно добавить периодический cron (бот раз в день проверяет всех active boosty-подписчиков), но пока не делаем — слишком много getChatMember-вызовов.
